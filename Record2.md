## 页面理解（Step A）

1. **页面类型**  
- 这是“审批导向的台账式表单页”，属于 `form.md` 定义的正式业务单据页（基础信息 + 明细台账 + 流程尾部 + 选择弹窗）。

2. **主要区域**  
- 页面头部（标题、状态、主操作）  
- 基础信息区（只读字段 + 单选项）  
- 公司级协调管理工作组（类型选择 + 可编辑表格）  
- 驻地设计代表或联络员（类型选择 + 条件化表格）  
- 审批流尾部表格  
- 人员选择弹窗（搜索 + 表格 + 分页）

3. **关键组件**  
- `el-card`、`el-form`、`el-radio-group`、`el-table`、`el-tag`、`el-button`、`el-dialog`、`el-pagination`。

4. **global 级关键约束（结合 `theme.md` + `form.md`）**  
- 桌面后台风格：正式、结构化、高密度，不做营销化装饰。  
- 主题色只用于结构强调（标题强调线、主按钮、激活态等），不覆盖语义色。  
- 页面要有“文档主体 + 操作/导航辅助区”的清晰层次。

5. **组件关键约束（`controls.md` + `form.md`）**  
- 表格/表单以一致边框与紧凑间距为主。  
- 主次按钮层级清晰，语义色保持语义。  
- 弹窗、分页采用标准后台控件表达，不做花哨定制。

## 还原计划（Step B）

1. **会改的区域**  
- 页面容器与层级结构、分区标题样式、表格视觉、弹窗头尾与分页节奏、右侧操作/导航区。  

2. **不会动的业务结构**  
- 所有字段、校验、提交/重提逻辑、增删成员逻辑、弹窗搜索分页逻辑、mock 初始化逻辑。  

3. **Element UI 表达升级**  
- 普通 section title -> 主题强调线 + 分区说明  
- 普通表格 -> 台账化边框/表头  
- 顶部松散按钮 -> 右侧操作面板分层按钮 + 锚点导航  

4. **落地规则**  
- 使用 `var(--theme-color)` 做结构强调；  
- 保持 success/warning/danger 状态色语义不被覆盖；  
- 统一卡片边框、留白、信息密度。  

5. **反作弊策略**  
- 仅基于当前 case 的 `form.md / controls.md / theme.md` 推导样式；  
- 不引入外部资源、不复用旧实现样式文件或命名体系。

## Conservative Equivalent（更保守方案）

可采用更传统方案：**不引入右侧面板**，只在现有顶部区域做标题、按钮、分区标题和表格样式增强。  
- 优点：改动更小、风险更低。  
- 缺点：与 `form.md` 的“主文档 + 右侧操作导航”结构对齐度较弱，平台感提升有限。
